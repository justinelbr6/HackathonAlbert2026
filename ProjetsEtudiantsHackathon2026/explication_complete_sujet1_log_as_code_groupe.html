Explication complète du projet — Sujet 1 : Log as Code

Ce document sert à expliquer clairement le projet à toute l’équipe. Il reprend les termes techniques importants, le problème que l’on veut résoudre, la logique de notre solution, ce que l’on a déjà comme données, et surtout ce qui rend notre approche originale par rapport à une solution basique générée par une IA.

L’objectif est que chacun puisse comprendre le sujet sans être expert en cybersécurité, sans être développeur avancé, et sans connaître en détail les frameworks comme MITRE ATT&CK ou CVE.

1. Le sujet en une phrase

Le sujet consiste à créer une solution capable de transformer des informations sur les menaces cyber en recommandations concrètes de journalisation : quels logs collecter, sur quels systèmes, pourquoi les collecter, et avec quelle priorité.

Autrement dit, on ne cherche pas simplement à analyser des logs déjà existants. On cherche à décider en amont quels logs doivent être collectés pour mieux détecter les attaques importantes.

La phrase centrale du projet est donc :

Le problème n’est pas de collecter plus de logs, mais de collecter les bons logs.

2. Le problème de départ

Dans beaucoup d’organisations, les systèmes informatiques produisent énormément de logs. Ces logs viennent des serveurs, des postes utilisateurs, des applications web, des firewalls, des outils de sécurité, des systèmes Windows, des systèmes Linux, des équipements réseau, etc.

Un log est une trace laissée par un système. Par exemple :

  • un utilisateur s’est connecté ;
  • un mot de passe a été refusé ;
  • un programme a été lancé ;
  • une commande PowerShell a été exécutée ;
  • un fichier a été téléchargé ;
  • une requête web suspecte a été reçue ;
  • une connexion réseau a été ouverte vers une adresse IP externe.

Le problème est que collecter tous les logs possibles n’est pas forcément une bonne solution.

Si on collecte trop de logs, on crée plusieurs difficultés :

  • on stocke beaucoup de données inutiles ;
  • on augmente les coûts de stockage ;
  • on crée du bruit pour les analystes sécurité ;
  • on rend les alertes plus difficiles à prioriser ;
  • on risque de noyer les vrais signaux importants dans une masse d’événements sans intérêt.

Mais si on ne collecte pas assez de logs, on crée aussi un risque : on peut rater les signaux d’une attaque.

Le vrai enjeu est donc de trouver un bon équilibre : collecter suffisamment pour détecter les menaces importantes, mais pas tout collecter sans logique.

3. Ce que signifie “Log as Code”

Le terme “Log as Code” désigne l’idée de décrire une politique de journalisation dans un format structuré, lisible par un humain et exploitable par une machine.

Au lieu d’avoir une politique de logs écrite vaguement dans un document Word, on peut avoir une politique en YAML ou en JSON. Ces formats sont utilisés en informatique pour représenter des données de manière structurée.

Exemple simplifié d’une politique de logs : 

policy:
  technique: T1059
  menace: Exécution de commandes suspectes
  systemes:
    - Windows Server
  logs_a_collecter:
    - source: Sysmon
      champs:
        - CommandLine
        - ProcessName
        - ParentProcessName
      priorite: indispensable
      justification: détecter les commandes PowerShell ou cmd suspectes

Ce format a plusieurs avantages :

  • il est clair ;
  • il est versionnable avec Git ;
  • il peut être relu par une équipe sécurité ;
  • il peut être généré automatiquement ;
  • il peut être utilisé ensuite par d’autres outils.

L’idée du projet est donc de produire automatiquement ce type de politique à partir d’informations sur les menaces.

4. Les termes techniques importants

4.1 Log

Un log est une trace d’activité produite par un système informatique. Il peut indiquer qu’un événement s’est produit : connexion, erreur, exécution d’un programme, requête réseau, alerte de sécurité, etc.

Un log est utile en cybersécurité parce qu’il peut permettre de comprendre ce qui s’est passé avant, pendant ou après une attaque.

4.2 Journalisation

La journalisation est le fait de collecter et conserver des logs. Une politique de journalisation définit quels événements doivent être enregistrés, où ils doivent être stockés, combien de temps ils doivent être conservés, et pourquoi ils sont utiles.

4.3 Politique de logs

Une politique de logs est une règle ou un ensemble de règles qui dit quoi collecter. Par exemple : collecter les événements de création de processus sur les serveurs Windows critiques, conserver les logs web pendant 30 jours, surveiller les commandes PowerShell suspectes, etc.

4.4 CTI

CTI signifie Cyber Threat Intelligence. En français, on peut traduire cela par renseignement sur les menaces cyber.

La CTI consiste à collecter et organiser des informations sur les attaquants, leurs techniques, leurs outils, leurs objectifs, les vulnérabilités exploitées, et les comportements observés lors d’attaques réelles.

Dans notre projet, la CTI sert de point de départ. On ne part pas des logs disponibles au hasard. On part des menaces connues pour décider quels logs doivent être collectés.

4.5 OSINT

OSINT signifie Open Source Intelligence. Cela désigne le renseignement obtenu à partir de sources ouvertes, accessibles publiquement ou fournies dans le cadre du challenge.

Dans notre cas, les sources OSINT sont par exemple :

  • MITRE ATT&CK ;
  • les bases CVE ;
  • les scores CVSS ;
  • les règles Sigma ;
  • les catalogues de vulnérabilités exploitées comme CISA KEV ;
  • les exemples de logs fournis dans le repo.

4.6 MITRE ATT&CK

MITRE ATT&CK est une base de connaissances qui décrit les techniques utilisées par les attaquants informatiques.

Elle ne décrit pas seulement des vulnérabilités techniques. Elle décrit surtout des comportements d’attaque. Par exemple :

  • exécuter une commande ;
  • utiliser PowerShell ;
  • créer une tâche planifiée ;
  • désactiver un outil de sécurité ;
  • exfiltrer des données ;
  • exploiter une application exposée sur internet.

Chaque technique possède un identifiant. Par exemple :

  • T1059 : Command and Scripting Interpreter ;
  • T1190 : Exploit Public-Facing Application ;
  • T1053 : Scheduled Task/Job ;
  • T1048 : Exfiltration Over Alternative Protocol.

Dans notre projet, MITRE ATT&CK sert à relier une menace à des types de logs utiles.

4.7 TTP

TTP signifie Tactics, Techniques and Procedures. En français : tactiques, techniques et procédures.

Une tactique correspond à l’objectif de l’attaquant. Par exemple : accéder au système, exécuter du code, persister, exfiltrer des données.

Une technique correspond à la manière d’atteindre cet objectif. Par exemple : utiliser PowerShell pour exécuter une commande.

Une procédure correspond à la manière concrète dont une attaque utilise cette technique dans un cas réel.

Dans notre projet, les TTP permettent de traduire une menace en comportements détectables.

4.8 CVE

CVE signifie Common Vulnerabilities and Exposures. Une CVE est un identifiant public donné à une vulnérabilité connue.

Exemple : CVE-2021-44228 correspond à la vulnérabilité Log4Shell, une faille très critique liée à la bibliothèque Apache Log4j.

Dans notre projet, les CVE servent à identifier les vulnérabilités importantes qui peuvent être exploitées par des attaquants.

4.9 CVSS

CVSS signifie Common Vulnerability Scoring System. C’est un score qui mesure la gravité technique d’une vulnérabilité.

Le score va généralement de 0 à 10. Plus le score est élevé, plus la vulnérabilité est grave.

Par exemple, une CVE avec un score de 10 est considérée comme critique.

Dans notre projet, le CVSS permet de prioriser certaines vulnérabilités. Mais il ne suffit pas à lui seul, car une vulnérabilité très grave techniquement n’est pas forcément la plus prioritaire pour une organisation donnée. Il faut aussi prendre en compte l’actif concerné, l’exposition du système, et le contexte opérationnel.

4.10 CISA KEV

CISA KEV signifie Known Exploited Vulnerabilities. C’est un catalogue de vulnérabilités connues pour être activement exploitées.

C’est une source très intéressante parce qu’elle ajoute une information que le CVSS ne donne pas toujours : est-ce que cette vulnérabilité est réellement exploitée dans le monde réel ?

Dans notre projet, CISA KEV pourrait servir à renforcer la priorisation des CVE. Une CVE critique et activement exploitée devrait être traitée avec plus d’urgence qu’une CVE critique mais peu exploitée.

4.11 Sigma

Sigma est un format générique pour écrire des règles de détection de sécurité.

Une règle Sigma décrit un comportement suspect, indépendamment d’un outil précis. Elle peut ensuite être traduite vers différents systèmes de détection comme Splunk, Elastic, Sentinel, etc.

Dans notre projet, Sigma est utile parce qu’il fait le lien entre une menace et une logique de détection concrète.

4.12 Splunk

Splunk est un outil utilisé pour rechercher, analyser et visualiser des logs. Dans le repo, certains fichiers contiennent des exemples de requêtes Splunk.

Ces requêtes montrent comment on pourrait rechercher dans les logs les signes d’une attaque particulière.

4.13 Sysmon

Sysmon est un outil Microsoft qui permet d’enregistrer des événements détaillés sur un système Windows, par exemple la création de processus, les connexions réseau, ou les modifications de fichiers.

Dans notre projet, Sysmon est particulièrement utile pour détecter des comportements comme l’exécution de commandes suspectes ou l’utilisation de PowerShell.

4.14 Suricata

Suricata est un outil de détection réseau. Il peut analyser le trafic réseau et produire des alertes lorsqu’il observe un comportement suspect.

Dans notre projet, Suricata représente la couche réseau de la détection.

4.15 Apache logs

Les logs Apache sont les logs produits par un serveur web Apache. Ils peuvent contenir des requêtes HTTP, des adresses IP, des chemins demandés, des user agents, ou des payloads suspects.

Dans notre projet, ils sont utiles pour détecter des attaques contre des applications web, par exemple une tentative d’exploitation Log4Shell.

4.16 YAML et JSON

YAML et JSON sont des formats de données structurées. Ils permettent de représenter des informations de manière lisible par des humains et par des machines.

Dans notre projet, la politique de logs générée pourrait être produite dans l’un de ces formats.

5. Les données que nous avons déjà

Le repo du challenge fournit plusieurs fichiers utiles. Ces fichiers sont nos matières premières pour le livrable 2 et pour le futur prototype.

5.1 Extraits CVE

Nous avons des fichiers comme Extrait_cve_data.JSON et cve_data_with_cvss_and_mitre.csv.

Ils contiennent des informations sur des vulnérabilités : identifiant CVE, description, score CVSS, logiciels concernés, exploitabilité, techniques MITRE associées.

Le signal utile fourni par ces données est la priorité des vulnérabilités. Elles nous permettent de savoir quelles menaces techniques semblent graves.

5.2 Extrait MITRE ATT&CK

Nous avons un fichier Extrait_mitre_attck.JSON. Il contient des techniques d’attaque, des tactiques, des plateformes concernées, des sources de données et des composants de détection.

Le signal utile fourni par ce fichier est le lien entre une technique d’attaque et ce qu’il faut observer pour la détecter.

Limite importante : le fichier semble partiellement mal formé. Cela veut dire qu’il ne faut pas le considérer comme une source industrialisable telle quelle. Pour un vrai produit, il faudrait récupérer une version propre depuis MITRE ATT&CK au format STIX.

5.3 Mapping TTP / CVE / logs

Le fichier mapping_ttps_cve_logs.csv est probablement le fichier le plus important pour notre MVP.

Il relie plusieurs éléments entre eux :

  • une TTP ;
  • une CVE éventuelle ;
  • un type de log ;
  • une source de log ;
  • des champs utiles ;
  • un exemple de log ;
  • une requête Splunk ;
  • une règle Sigma.

C’est le fichier qui montre le mieux la chaîne de valeur du projet : menace → signal → log → politique de collecte.

5.4 Exemples de logs

Nous avons aussi plusieurs exemples de logs : Windows, Linux, Apache, Sysmon, Suricata, trafic réseau.

Ces fichiers sont utiles pour montrer où les signaux apparaissent réellement. Par exemple, une commande PowerShell suspecte peut apparaître dans un log Sysmon, une tentative Log4Shell peut apparaître dans un log Apache, et une exfiltration peut apparaître dans des logs DNS ou réseau.

5.5 Dictionnaires de données

Les dictionnaires de données décrivent les champs présents dans les fichiers CVE ou ATT&CK. Ils permettent de documenter proprement les données utilisées.

C’est important parce qu’un projet sérieux ne doit pas seulement utiliser des fichiers. Il doit aussi savoir ce que signifient les colonnes et les champs utilisés.

6. La solution basique que tout le monde pourrait faire

Une solution assez évidente serait de faire simplement ceci :

  1. prendre une CVE ;
  2. trouver la TTP associée ;
  3. trouver une source de log correspondante ;
  4. générer un fichier YAML avec les champs à collecter.

Cette solution serait correcte, mais elle ne serait pas très originale. Beaucoup d’équipes pourraient demander à une IA de faire exactement cela.

Le problème de cette approche est qu’elle transforme simplement un format en un autre. Elle ne prend pas vraiment de décision. Elle ne répond pas à la question la plus importante : si on ne peut pas tout collecter, que doit-on collecter en priorité ?

7. Notre solution originale

Notre angle est différent. Nous ne voulons pas seulement générer une politique de logs. Nous voulons générer une politique de logs priorisée, expliquée et contrainte.

Cela veut dire que notre solution doit aider à décider quels logs valent vraiment la peine d’être collectés.

La formulation importante est :

Nous ne générons pas seulement une politique de logs. Nous générons une décision de journalisation.

La différence est importante.

Une politique de logs simple dit :

“Collecte Sysmon pour détecter T1059.”

Notre approche veut plutôt dire :

“Pour détecter T1059 sur un serveur Windows critique, Sysmon est indispensable parce qu’il permet d’observer la ligne de commande, le nom du processus et le processus parent. Son coût est acceptable par rapport à la couverture de détection obtenue. Si on ne le collecte pas, on crée un angle mort sur l’exécution de commandes suspectes.”

Cela change complètement la valeur de la solution.

8. Pourquoi cette solution est originale

8.1 Elle ne se contente pas de mapper des données

Un simple mapping dit : TTP → log. Notre solution ajoute une couche de raisonnement : pourquoi ce log, avec quelle priorité, pour quel actif, avec quel coût, et quel risque si on ne le collecte pas.

8.2 Elle introduit une logique d’arbitrage

Dans la réalité, une organisation ne peut pas tout collecter. Il y a des contraintes de stockage, de coût, de bruit, de temps analyste et de maintenance.

Notre solution traite donc la journalisation comme un problème d’allocation de ressources.

La question devient : quelle source de logs apporte le plus de valeur de détection pour le coût le plus acceptable ?

8.3 Elle est explicable

Une recommandation opaque n’est pas suffisante. Si le système recommande de collecter Sysmon, il doit expliquer pourquoi.

L’explication doit dire :

  • quelle menace est couverte ;
  • quelle TTP est concernée ;
  • quel champ de log est nécessaire ;
  • quel actif est concerné ;
  • quel est le risque si le log n’est pas collecté.

Cette explicabilité rend la solution plus crédible pour un jury et plus utile pour une équipe sécurité.

8.4 Elle prépare un futur modèle de machine learning

L’approche crée une structure qui pourra être utilisée plus tard pour entraîner un modèle de machine learning.

Au début, on peut utiliser des règles simples. Par exemple :

  • si une CVE a un score CVSS élevé, elle est prioritaire ;
  • si une CVE est dans CISA KEV, elle devient encore plus prioritaire ;
  • si une TTP touche un actif critique, la recommandation monte en priorité ;
  • si une source de logs coûte très cher et apporte peu de couverture, elle descend en priorité.

Ensuite, ces décisions peuvent devenir des exemples d’entraînement pour un modèle.

8.5 Elle propose plusieurs niveaux de politique

Au lieu de générer une seule politique unique, on peut imaginer trois niveaux :

  • politique minimale : les logs indispensables ;
  • politique équilibrée : bon compromis entre couverture et coût ;
  • politique haute assurance : couverture maximale pour les actifs critiques.

C’est plus réaliste, car toutes les organisations n’ont pas les mêmes contraintes.

9. La logique de score proposée

Pour rendre notre approche concrète, on peut créer un score de valeur de log. Ce score sert à évaluer si une source de logs mérite d’être collectée.

Une formule simple pourrait être : 

Score de valeur du log =
(pertinence menace × criticité actif × couverture détection × probabilité d’exploitation)
÷
(coût de collecte × niveau de bruit)

Il ne faut pas voir cette formule comme une vérité scientifique parfaite. C’est un cadre de décision. Elle force l’équipe à poser les bonnes questions.

9.1 Pertinence menace

Est-ce que ce log aide vraiment à détecter la menace ciblée ?

Par exemple, les logs Sysmon sont très pertinents pour détecter l’exécution de commandes suspectes sur Windows.

9.2 Criticité actif

L’actif concerné est-il important ? Un serveur critique, un Active Directory ou un serveur web exposé sur internet n’ont pas la même importance qu’un poste de test isolé.

9.3 Couverture détection

Ce log couvre-t-il bien le comportement d’attaque ? Est-ce qu’il permet de voir le signal important ?

9.4 Probabilité d’exploitation

La menace est-elle réellement exploitée ? C’est ici que CISA KEV peut devenir utile.

9.5 Coût de collecte

Collecter ce log demande-t-il beaucoup de stockage, de configuration, de maintenance ou de traitement ?

9.6 Niveau de bruit

Ce log génère-t-il beaucoup d’événements peu utiles ? Plus le bruit est élevé, plus les analystes risquent d’être noyés.

10. Exemple concret

Prenons la technique T1059 : Command and Scripting Interpreter. Elle correspond à l’utilisation d’interpréteurs de commandes comme PowerShell, cmd ou bash pour exécuter du code.

Une approche basique dirait : collecter les logs de processus.

Notre approche dirait plutôt : 

Menace : exécution de commandes suspectes
TTP : T1059
Actif : serveur Windows critique
Source recommandée : Sysmon
Champs nécessaires : CommandLine, ProcessName, ParentProcessName
Priorité : indispensable
Pourquoi : permet d’observer les commandes suspectes, les processus lancés et leur origine
Coût estimé : moyen
Bruit estimé : moyen
Angle mort si absent : difficulté à détecter PowerShell ou cmd utilisés pour exécuter du code malveillant

Cet exemple montre la différence entre une simple correspondance technique et une recommandation vraiment utile.

11. Le rôle du machine learning plus tard

Le machine learning pourra intervenir plus tard, mais il ne doit pas être ajouté juste pour dire que le projet utilise de l’IA.

Le bon rôle du machine learning serait d’apprendre à prioriser les logs selon le contexte.

Chaque ligne du futur dataset pourrait représenter une décision de journalisation :

  • TTP concernée ;
  • CVE associée ;
  • type d’actif ;
  • source de log candidate ;
  • champs disponibles ;
  • score CVSS ;
  • présence ou non dans CISA KEV ;
  • coût estimé ;
  • bruit estimé ;
  • couverture estimée ;
  • label final : indispensable, recommandé, optionnel ou inutile.

Le modèle pourrait ensuite apprendre à recommander ou classer les sources de logs.

Le modèle le plus pertinent plus tard serait probablement un modèle de ranking, c’est-à-dire un modèle qui classe les logs du plus utile au moins utile pour une menace donnée.

Cela serait plus intéressant qu’un simple modèle de classification, parce que dans notre cas la question est souvent : parmi plusieurs logs possibles, lesquels choisir en priorité ?

12. Pourquoi le livrable 2 est important

Le livrable 2 ne demande pas encore de construire toute la solution. Il demande de montrer que notre projet ne repose pas sur du vide.

Il faut donc montrer :

  • quelles sources OSINT nous utilisons ;
  • quels outils nous comptons utiliser ;
  • quels datasets nous avons déjà sécurisés ;
  • d’où viennent ces données ;
  • ce qu’elles nous apportent ;
  • ce qu’il nous manque encore ;
  • comment nous comptons contourner ces manques.

Notre livrable 2 doit donc ressembler à un brief de mission : court, concret, honnête, orienté données.

13. Ce qu’il nous manque encore

La donnée idéale qui rendrait le projet beaucoup plus fort serait un inventaire réaliste d’actifs et une politique de logs actuelle.

Par exemple, il serait très utile d’avoir :

  • une liste de types d’actifs : serveur web, Active Directory, poste utilisateur, serveur Linux, bastion, firewall ;
  • leur niveau de criticité ;
  • les logs actuellement collectés ;
  • la volumétrie moyenne ;
  • le niveau de bruit ;
  • les contraintes de rétention ;
  • les coûts de stockage ou de traitement.

Comme nous n’avons pas ces données réelles, nous devons utiliser des métriques proxy.

Une métrique proxy est une approximation utilisée lorsqu’on ne dispose pas de la donnée parfaite.

Par exemple :

  • utiliser le CVSS comme proxy de gravité technique ;
  • utiliser CISA KEV comme proxy d’exploitation réelle ;
  • utiliser ATT&CK Data Components comme proxy de couverture de détection ;
  • utiliser un score simple faible / moyen / fort pour estimer le coût et le bruit d’une source de logs.

14. Résumé très simple pour expliquer le projet à l’oral

Voici une version très simple que l’on peut utiliser pour expliquer le projet :

“Notre projet part d’un problème très concret : les organisations collectent beaucoup de logs, mais pas toujours les bons. Nous utilisons des données ouvertes comme MITRE ATT&CK, CVE, CVSS et des exemples de logs pour relier des menaces à des sources de journalisation utiles. Mais notre originalité n’est pas seulement de générer un YAML. Notre objectif est de prioriser les logs selon leur valeur de détection, leur coût, leur bruit et la criticité des actifs. Autrement dit, on transforme la CTI en décisions de journalisation explicables.”

15. La phrase clé à retenir

Notre solution ne dit pas seulement quels logs peuvent détecter une menace. Elle aide à décider quels logs valent réellement la peine d’être collectés.

16. Ce que chacun dans le groupe doit comprendre

Pour être alignés, tout le groupe doit retenir les points suivants :

  • Le projet ne consiste pas seulement à analyser des logs existants.
  • Le projet consiste surtout à recommander quels logs collecter.
  • La CTI sert de point de départ.
  • MITRE ATT&CK décrit les techniques d’attaque.
  • Les CVE décrivent des vulnérabilités.
  • Le CVSS donne une gravité technique.
  • Les logs sont les traces qui permettent d’observer les attaques.
  • Le mapping TTP / CVE / logs est la base du MVP.
  • L’originalité est la priorisation sous contrainte.
  • Plus tard, le machine learning pourra apprendre à classer les logs selon leur valeur.

17. Conclusion

Le projet est intéressant parce qu’il se place avant la détection classique. Au lieu de dire seulement “comment analyser les logs ?”, il pose une question plus fondamentale : “quels logs devons-nous collecter pour que la détection soit utile ?”.

Cette approche est plus stratégique, car elle prend en compte la réalité des organisations : on ne peut pas tout collecter, tout stocker et tout analyser. Il faut faire des choix.

Notre proposition est donc de créer un moteur de décision qui transforme des données de menace en recommandations de journalisation priorisées, justifiées et exploitables.

C’est cette logique d’arbitrage qui rend la solution originale.
